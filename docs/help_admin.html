1. What the app does & the one rule that governs everything

The app turns three Amazon exports plus one human-maintained stock workbook into a per-product, per-region restocking plan for an Amazon India FBA seller. It works out sales velocity, assembles current + in-transit + at-FC stock, models the BLR4 cross-dock (IXD), rounds each shipment to carton sizes, simulates stockouts, and writes one Excel plan.

THE governing constraint — internalise this first: the app is stateless. Streamlit Community Cloud spins it up and down; it has no database and no memory between runs. The GitHub repository IS the only persistent store. Every standing fact — box sizes, FC→region mapping, config numbers, region splits, demand-state mapping — lives in a committed file and is re-read fresh on every run. If a change isn't committed to the repo, it does not exist to the next run. This single fact explains almost every design decision below.
Corollary you will feel in practice: because it is stateless, the app re-runs the entire pipeline — including re-parsing the registration PDF and the sales CSV — on every file upload and every widget click. A full pass is several seconds locally and can be longer on Cloud hardware, so after each upload the app may sit on a spinner for 10–30 s. That is recomputation, not a freeze.

3. Architecture — the five-module pipeline

The app is a strict, gated pipeline. Each module owns one job and is independently testable. The app file only orchestrates and renders — no business logic lives there.

4. Business logic, end to end — the calculation step by step

This is the heart of the app: how a shipped quantity for one product in one region is decided. Every formula below is stated in plain English first, then exactly as the code computes it. All date arithmetic anchors on the stock anchor date (the ledger's data date), never on today. The engine keys everything on ASIN, not SKU, and writes the answer back to the SKU at the very end.

STEP 0Derive the regions from the registration. (ingestion.load_fc_registration)

Each fulfilment centre (FC) is read from the registration file as a (2-letter state code, FC code) pair and mapped to a region label. An FC listed IGNORE in FC_Types becomes YSXA; one listed IXD becomes BLR4 IXD. A split state (Region_Splits) sends each FC to the sub-region whose prefix it matches (e.g. PNQ3 → Pune (MH)), otherwise to that state's DEFAULT region; every other state becomes Capital (CODE), e.g. Delhi (DL). In the current data, 40 FCs collapse to 8 states → 9 destination columns (MH splits into Bombay + Pune) plus BLR4 IXD and YSXA.

STEP 1Send each demand state to the region that serves it. (calculation.resolve_demand_regions)

The sales file gives demand per customer shipping state. Resolution order: normalise the spelling → if the state has its own FCs, it serves itself (a split state resolves to its DEFAULT, so MAHARASHTRA → Bombay (MH)) → else look it up in Demand_Map for its serving region → else it is EXCLUDED and named with its unit count (never silently dropped).

The boundary rule keeps the two mechanisms from overlapping: the registration handles "state with an FC → itself" automatically; Demand_Map handles only "state with no FC → serving region." A Demand_Map row naming a state that already has FCs is an error by design.
STEP 2Turn the sales window into a daily rate. (ingestion.load_sales → calculation.pool_demand)

The window length is inferred from the earliest and latest Shipment Date in the file (or supplied via the sidebar override for a date-less file). Sales are always pooled by ASIN across all its SKUs.

daily rate = window units ÷ sales_window_days (summed per ASIN × region)
STEP 3Assemble what is already available or on the way. (calculation.assemble_supply)

Supply is read only from the completed stock workbook (the sole planning-stock input). Three components per ASIN × region:

  • Current stock — on-hand for the planning regions. Stock parked at the hub is set aside as the IXD pool; under an "only-active" ASIN judgment the suppressed SKU's stock is stranded (excluded-not-dropped).
  • In-transit counted — an in-transit line counts only if its committed reach date lands inside the region's horizon (lead + days_of_cover). Lines landing later are ignored this run.
  • At-FC pending — units physically at the FC but not yet received, counted at day 0 and flagged. pending = shipped − received; received is never re-counted because it is already inside Current stock.
STEP 4Redistribute the cross-dock stock. (calculation.apply_ixd)

Stock at the BLR4 IXD hub has no region of its own; Amazon spreads it to regional FCs. The engine splits each ASIN's hub stock across its demand regions in proportion to each region's daily rate, scaled by ixd_confidence. If an ASIN has no demand anywhere, its hub stock stays put.

ixd_offset(region) = hub_qty × ixd_confidence × daily(region) ÷ Σ daily(all regions)

In the stockout sim this offset is treated as arriving after ixd_transfer_days.

STEP 5Compute the raw requirement. (calculation.raw_requirements)

Only regions that have a lead time are planned. Forecast the depletion over lead + cover, subtract everything available, floor at zero.

depletion = daily × (lead + days_of_cover) available = current + in_transit_counted + at_fc_pending + ixd_offset raw requirement = max(0, depletion − available)
STEP 6Round to shippable cartons (two tiers). (calculation.round_boxes)

box = the ASIN's Units/Box from the master. Let carton_days = box ÷ daily.

  • Tier 2 — slow mover: if a single carton would last longer than days_cover_ceiling at this rate, ship a partial quantity instead of forcing a whole carton. Flag PARTIAL_BOX.
    if carton_days > days_cover_ceiling → qty = ceil(raw requirement)
    (Assumption A1: it ships ceil(raw) units, not an inner-pack multiple, because the master has no inner-pack column yet.)
  • Tier 1 — default: round up to a full carton. Flag FULL_CARTON.
    qty = ceil(raw requirement ÷ box) × box
  • Round-down guard: a tiny need on a non-urgent line is dropped to zero rather than shipping a near-empty carton. Flag ROUNDED_DOWN.
    if raw < 0.30 × box and line is not Pri-1 → qty = 0
    (Assumption A2: the 0.30 fraction is a module constant, not yet a Config key.)

Resulting cover after the plan lands (shown on the Sales-velocity tab; can legitimately exceed your setting when a carton overshoots or existing stock already covers more):

days_cover_achieved = (current + in_transit + at_fc_pending + ixd_offset + qty) ÷ daily − lead
STEP 7Sanity-check with a day-by-day stockout walk. (calculation.simulate_stockouts)

Using the rounded quantities, walk each region day by day across the horizon. Start with current + at_fc_pending; add in-transit on its committed day, the IXD offset on ixd_transfer_days, and the planned shipment on the region's lead day; subtract the daily rate each day. Flag a stockout only if the shortfall drops below −min_stockout_deficit_units (the threshold kills sub-unit noise).

STEP 8Tag urgency (priorities). (calculation.compute_priorities)

Per line, "risk" is how many days short of the lead time the current stock will run:

risk = lead − (current ÷ daily) Pri-1 if risk ≥ 2 or current = 0 · Pri-2 if risk ≥ 0 Pri-3 if risk ≥ −3 · Pri-4 otherwise (daily = 0 → Pri-4)

A region's priority is set the same way from the mean risk of its lines (bands ≥1 / ≥−1 / ≥−3). Pri-1 is the most urgent.

The whole chain in one sentence

For each product × region: forecast demand over lead + cover, subtract everything already on hand or on the way (including a share of the cross-dock), round the gap up to full cartons unless a carton would overstock a slow mover, then day-walk the rounded plan to catch stockouts and tag each region's urgency. The ASIN answer is finally written to the SKU that carried the sales this window (assumption A4); split-state secondary columns (Pune) are left empty for the human to fill.

7. Configuration reference

configurations.xlsx is validated on load; a bad edit (blank number, unknown FC type, a split with zero or two DEFAULTs, an orphaned FC, an FC-state row in Demand_Map) fails loud with a named error and stops the run.

Config tab — the seven tunable numbers

KeyDefaultWhat it does / when to touch it
days_cover_ceiling45Slow-mover guard. If one full carton would last more than this many days at a location's rate, the app ships a partial box (Tier-2) instead of forcing the carton. Raise it to allow bigger single cartons; lower it to trim slow movers harder.
min_sales_window_days30Hard floor on sales-window length, for both date-inferred and override windows. Below it the daily rate is statistically noisy → a warning. Lower it only if you knowingly plan on short windows.
freshness_tolerance_days3Max spread among stock-type file dates (ledger / general / workbook). Beyond it → hard stop, because a stock picture assembled from different moments is corrupt.
ixd_transfer_days5Assumed days for BLR4 IXD stock to reach a regional FC after Amazon redistributes it. Times when the IXD offset "arrives" in the stockout sim.
ixd_confidence1.0A 0–1 dial on how much of the cross-dock stock to count as regional supply. 1.0 = trust the redistribution fully (count all of it); 0 = ignore hub stock for planning (treat it as informational only); 0.5 counts half. Lower it if hub stock often fails to arrive in time.
min_stockout_deficit_units1Stockout warnings smaller than this many units are suppressed, so you don't drown in sub-unit noise.
sales_recency_prompt_days v2.960If the sales window ends more than this many days before the stock date, the app asks you to confirm (not a hard stop) — deliberately planning on an old baseline is a judgment you own, acknowledged and flagged.
Config↔code coupling: every key in the code's CONFIG_KEYS (in ingestion_v2.py) is REQUIRED in the Config tab, and each value is range-checked. If you add a key in one place, add it in the other and commit both together, or you get a "missing key" hard error or an "unknown parameter (ignored)" warning.

8. Editing the reference files (walkthroughs)

Golden sequence for ANY reference-file change: download the current file from the repo → edit in Excel → upload back to the SAME path with the SAME filename → one-line commit → the app picks it up on the next run automatically. Never rename, never delete-then-reupload.

Add a new SKU (the most common admin task)

Trigger: a run halts with "STRICT SKU GATE — file contains SKU(s) not in the master template: [NEW-SKU]". By design — the app will not invent a box size.

  1. Download reference/inventory_plan_template.xlsx.
  2. On the Appointment plan sheet, add a row for the SKU inside the existing SKU block (no blank row inside it — parsing stops at the first blank).
  3. Fill every master column: SKU (exactly as Amazon spells it), Alias / Display name, ASIN (the planning key), ITEM (the human product name — this is what shows as "Product" in outputs, not the Alias, which repeats SKUs), Units/Box (carton size — drives all rounding), Bottles/Unit, Litres/Unit, Kg/Unit, Oil Type, Bottle Type.
  4. If it's a temporary/suppression SKU of an existing product, give it the SAME ASIN — the app pools them and asks the stock judgment per run.
  5. Mirror the row on the Stock snapshot sheet's block so the snapshot has somewhere to write.
  6. Upload in place, commit ("add SKU XYZ, 12/box"), rerun — the gate clears.
Units/Box is the single most consequential cell you will ever edit. A wrong carton size silently mis-rounds every shipment of that SKU. Check it against the physical carton.

Other edits, in brief

  • Tune a number → Config tab, column B only, commit. Column A key names are what the code looks up — never rename them.
  • Add a Demand_Map row (a no-FC state was EXCLUDED in a run) → spelled-out state name | 2-letter serving code (a state that HAS FCs) | note. Never add a state that has its own FCs.
  • Declare a special FC → FC_Types: FC code | exactly IXD or IGNORE. Ordinary new FCs need nothing.
  • Change the output layout → edit inventory_plan_template.xlsx and commit; the writer finds columns by header name, so keep header text identical.
  • Lead times are NOT in config — they live in the daily stock workbook's Reference tab, edited by the user each run.

9. The validation gates & their philosophy

Design philosophy, consistent throughout: fail loud, fail specific, never guess; where a fact needs human judgment, stop and ask rather than assume. The app plans only when hard errors are zero AND every interactive gate is answered. Gate order:

schema / reference errors     (hard stop)
   -> FC-review gate          (interactive, v2.9)  unregistered FC holding stock
   -> stock-vs-stock freshness (hard stop)
   -> sales-recency ack        (interactive, v2.9)  old sales window
   -> workbook read            (hard stop on a bad workbook)
   -> ASIN judgment gate       (interactive)        one ASIN under 2+ SKUs
   -> Calculate
GateTriggerResolution
Strict SKU gateA SKU in any Amazon file is absent from the masterHALT. Add the SKU to the master, commit, rerun. No partial plans.
FC-review v2.9Ledger stock in an FC not in the registration + FC_Types (the MAA4/GST case)Interactive: map-to-region / IXD / ignore / abort, + fulfillable? Per-run only; permanent fix = commit a refreshed registration PDF.
Stock freshnessStock-file dates spread > freshness_tolerance_daysHALT. Refresh the stale stock file.
Sales-recency v2.9Sales window ends > sales_recency_prompt_days before the stock dateAcknowledge & proceed (flagged). Legitimate for deliberate baselines.
ASIN judgmentAn ASIN holds stock under 2+ SKUs (suppression episode)Per-run: combine / only-active. The stranded SKU is excluded-not-dropped.

11. Where the deep documentation & tests live

  • The contract: Inventory_App_Redesign_Spec.md (v2.9) — every design decision, section-numbered, with a changelog per version. Read it before changing behaviour; update it FIRST, then the code.
  • Regression testing: golden_harness_v2.py — run python3 golden_harness_v2.py. 12. Glossary

Anchor date
The date all forward maths counts from — the ledger's data date (Amazon runs ~2 days behind live), never "today". Set on the workbook's "Stock as of" cell.
ASIN
Amazon's product identity. The app's true planning key: one product can wear two SKUs, so it groups by ASIN before doing demand, stock and box maths.
Excluded-not-dropped
Stock the app shows but deliberately doesn't count this run (e.g. a suppressed SKU's stock, or a not-fulfillable mapped FC). Visible and reversible, never silently lost.
FC (fulfilment centre)
An Amazon warehouse. Each FC belongs to a state; FCs in a state are pooled into one planning region with one lead time.
Horizon
The planning window for a region: lead + days_of_cover days from the anchor date. In-transit stock only counts if it lands inside this window.
IXD / cross-dock (BLR4)
An inbound cross-dock hub that holds no demand of its own; Amazon redistributes its stock to regional FCs. Counted in grand-total on-hand and split proportionally into regional supply, but never a shipping destination.
IXD confidence
A 0–1 setting for how much of the hub's stock to trust as usable regional supply. 1.0 = count it all; 0 = ignore it for planning. See ixd_confidence in §7.
Reach date
The committed date an in-transit shipment is expected to land at its FC — the only in-transit date the app reads. The workbook suggests one; the human commits the actual.
Split state / DEFAULT region
A state whose stock is sent to two destination regions (e.g. MH → Bombay + Pune) while demand is still measured for the whole state. Exactly one region is the DEFAULT and receives the full forecast; the other starts empty for manual filling.
Suppression / Temp-SKU
When Amazon suppresses a listing, a temporary SKU is created under the same ASIN. Sales are always pooled; whether to also pool the two SKUs' stock is the per-run "combine / only-active" judgment.
Tier-1 / Tier-2 rounding
Tier-1 rounds a requirement up to a full carton. Tier-2 (slow movers) ships a partial quantity when a whole carton would last past days_cover_ceiling.
Velocity
Daily sales rate = window units ÷ window days. The number that drives every region's requirement; shown beside the plan on the Sales-velocity tab.
