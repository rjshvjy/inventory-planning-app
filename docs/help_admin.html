1. What the app does & the one rule that governs everything

The app turns three Amazon exports plus a human-maintained stock workbook into a per-SKU, per-region replenishment plan for an Amazon India FBA seller (Puvi). It computes sales velocity, assembles current + in-transit + at-FC stock, models the BLR4 cross-dock (IXD), rounds to carton sizes, simulates stockouts, and writes an Excel plan.

THE governing constraint — internalise this first: the app is stateless. Streamlit Community Cloud spins the app up and down; it has no database and no memory between runs. The GitHub repository IS the only persistent store. Every standing fact — box sizes, FC→region mapping, config numbers, region splits, demand-state mapping — lives in a committed file and is re-read fresh on every run. If a change isn't committed to the repo, it does not exist to the next run. This single fact explains almost every design decision below.

3. Architecture — the five-module pipeline

The app is a strict, gated pipeline. Each module is independently testable and owns one job. The app file (Inventory_planning_app_v2.py) only orchestrates and renders — no business logic lives there.

Module responsibilities

Module~linesOwns
ingestion_v2.py751Reading & validating every input file; content-based column detection; FC→region mapping from the registration; sales pooling to sku×state; the strict SKU gate; the interactive-gate detection (FC-review, sales-recency). Produces Canonical + Report.
workbook_builder_v2.py843BUILDING the daily stock workbook fresh from code (all tabs, formulas, enforced dropdowns, styling) AND READING a filled one back into StockInputs. The one module the app both writes with and reads with.
calculation_v2.py549The planning engine. Pure functions, no Streamlit. Demand state→region resolution (§5b), supply assembly, IXD model, two-tier box rounding, stockout simulation, priorities. Produces PlanResult.
output_v2.py367Writing the plan Excel by cloning the master template and filling it; the Calculation_Details, Stock snapshot and Sales-velocity tabs; all styling harvested from the template.
Inventory_planning_app_v2.py301Streamlit orchestration only: uploads, the gate sequence, rendering results, download buttons. No business logic.
golden_harness_v2.py278Headless test harness — runs the whole pipeline on sample data, dumps a comparison workbook, asserts known-good numbers. The entry point for regression testing.

The two data contracts (the objects that flow between modules)

Canonical (from ingestion) carries: config, fc_types, region_splits, demand_map, fc_resolutions, fc_region, regions, sku_master, sales_daily, sales_window_days, national, available, effective_dates.

StockInputs (from the workbook reader) carries: current, as_on_date, in_transit, at_fc, lead_days, stock_as_of.

When editing any module, treat these dataclass field sets as the contract. Adding/removing a field ripples to every consumer — grep the field name across all modules before changing it. (This is exactly how a past mislabelling bug crept in.)

Given statelessness (§1), the repo layout is the app's memory:

Update discipline — non-negotiable: to change a reference file, replace it in place with the SAME filename (GitHub "upload files", drag it in, commit). NEVER delete-then-reupload: a renamed file (configurations (1).xlsx) will not be found by the app, and deletion fragments the git history you'd roll back through. One-line commit message describing the change.

5. Configuration reference

configurations.xlsx is validated on load; a bad edit (blank number, unknown FC type, a split with zero or two DEFAULTs, an orphaned FC, an FC-state row in Demand_Map) fails loud with a named error and stops the run. Required tabs: Config, FC_Types, Region_Splits, Demand_Map (plus a Help tab, free-text).

Config tab — the seven tunable numbers

KeyDefaultWhat it does
days_cover_ceiling45Slow-mover guard: if one full carton exceeds this many days of cover at a location's rate, propose a partial box (Tier-2) instead of forcing the carton.
min_sales_window_days30HARD floor on sales-window length (applies to both date-inferred and override-supplied windows). Below this the rate is statistically noisy → error. Remedy is editing this key, not code.
freshness_tolerance_days3Max spread among STOCK-type file dates (ledger / general / workbook). Beyond it → hard stop (a stock picture from different moments is corrupt).
ixd_transfer_days5Assumed days for BLR4 IXD stock to reach a regional FC after Amazon redistributes. Times the IXD-offset arrivals.
ixd_confidence1.00–1. Fraction of the IXD proportional-split offset to apply. 1.0 = trust the split fully; 0 = treat IXD stock as informational only.
min_stockout_deficit_units1Stockout warnings below this many units are suppressed (kills sub-unit noise).
sales_recency_prompt_days v2.960If the sales window's END is more than this many days behind the stock anchor, the app ASKS the user to confirm (not a hard stop). Deliberately using an old baseline is a judgment, acknowledged & flagged.
Config↔code coupling: a key listed in the code's CONFIG_KEYS is REQUIRED in the Config tab. If you add a key to the tab, add it to CONFIG_KEYS in ingestion_v2.py too (and vice-versa), or you get either a "missing key" hard error or an "unknown parameter (ignored)" warning. When deploying a code change that adds a key, commit the updated configurations.xlsx in the same push.

FC_Types tab

Lists ONLY non-ordinary FCs. Type must be exactly IXD or IGNORE (a typo like IDX stops the run). Every FC not listed is treated as an ordinary regional FC automatically — so this tab stays 2–3 rows (currently BLR4=IXD, YSXA=IGNORE).

Region_Splits tab

Declares a state whose stock is SENT to more than one destination region (e.g. MH → Bombay + Pune) while demand is still measured for the whole state. Exactly one region per split state must be marked DEFAULT (gets the full forecast); others start empty for manual filling. Validation enforces one-and-only-one DEFAULT and no orphaned FCs.

Demand_Map tab v2.8

Maps demand states WITHOUT a local FC to the region that serves them (e.g. Kerala→Bangalore). Boundary rule: the registration handles "state with FC → itself" automatically; Demand_Map handles ONLY "state without FC → serving region." A row naming an FC state is an error. A demand state that is neither an FC state nor mapped becomes a named warning with its unit count (never a silent drop).

6. The validation gates & their philosophy

Design philosophy, consistent across the app: fail loud, fail specific, never guess; where a fact needs human judgment, stop and ask rather than assume. The app proceeds to planning only when hard errors are zero AND every interactive gate is answered. Gate order:

schema/reference errors  (hard stop)
   -> FC-review gate      (interactive, v2.9)  unregistered FC holding stock
   -> stock-vs-stock freshness  (hard stop)
   -> sales-recency ack   (interactive, v2.9)  old sales window
   -> workbook read       (hard stop on bad workbook)
   -> ASIN judgment gate  (interactive)        one ASIN under 2+ SKUs
   -> Calculate
GateTriggerResolution
Strict SKU gateA SKU in any Amazon file is absent from the masterHALT. Add the SKU to the master, commit, rerun. No partial plans.
FC-review v2.9Ledger stock in an FC not in the registration + FC_Types (the MAA4/GST case)Interactive: map-to-region / IXD / ignore / abort, + fulfillable? Per-run only; permanent fix = commit a refreshed registration PDF.
Stock freshnessStock-type file dates spread > freshness_tolerance_daysHALT. Refresh the stale stock file.
Sales-recency v2.9Sales window ends > sales_recency_prompt_days before stock dateAcknowledge & proceed (flagged). Legitimate for deliberate baselines.
ASIN judgmentAn ASIN holds stock under 2+ SKUs (suppression episode)Per-run: combine / only-active. Excluded-not-dropped for the stranded SKU.

7. Maintenance scenarios

When this happens……the admin does this
A new SKU starts sellingAdd its row to inventory_plan_template.xlsx (SKU, ASIN, Units/Box, per-unit data), commit. Until then the strict gate halts any run containing it (by design).
An unregistered FC shows stock (MAA4-style)Short-term: the user answers the FC-review gate per run. Permanent: get the refreshed registration PDF from Seller Central, commit it in place — the gate then stops firing.
A new FC appears in the registrationNothing, usually — an ordinary regional FC is picked up automatically. Only if it's a cross-dock or warehouse does it need an FC_Types row.
A new demand state without an FC appears (named warning)Add a Demand_Map row (state → serving region), commit.
Amazon reformats an exportIngestion locates columns by identity (header name, then content shape), so header renames/reorders survive. A change in a column's MEANING is caught by the known-answer FC check and, ultimately, a human. Re-run the golden harness to confirm.
A Config number needs tuning (e.g. lower the noisy-rate floor)Edit the value in the Config tab, commit. No code change, no redeploy needed beyond the auto-redeploy.
A lead time changedLead times live in the daily stock workbook's Reference tab (per-run editable), not in config — the user updates it when filling the workbook.
The output/template layout needs a column movedEdit inventory_plan_template.xlsx in Excel and commit; the writer finds columns by header NAME, so it adapts with no code change (keep header text identical).

8. Where the deep documentation & tests live

  • The contract: Inventory_App_Redesign_Spec.md (v2.9) — every design decision, section-numbered, changelog per version. Read this before changing behaviour; update it FIRST, then the code.
  • Regression testing: golden_harness_v2.py — run python3 golden_harness_v2.py. It rebuilds fixtures, runs the full pipeline on the sample July-2026 data, dumps golden_out/golden_comparison.xlsx, and asserts any values filled into its GOLDEN dict. After any engine change, run it and confirm the numbers hold.
  • Handover trail: the HANDOVER_next_chat_*.md series records how the build progressed and why each decision was made — useful archaeology when a choice looks puzzling.
Golden rule for any change: spec first → code → run the harness → commit. Never let the code and the spec diverge; the spec is what a future maintainer (or a fresh session) trusts.
